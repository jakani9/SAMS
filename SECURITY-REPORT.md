# SAMS セキュリティ診断レポート

**対象:** `juku-attendance`（SAMS）  
**実施日:** 2026年7月4日  
**方式:** ソースコードレビュー、起動構成の確認、攻撃シナリオ分析、HTTP サーバー動作確認

---

## 総合評価

| 項目 | 評価 |
|------|------|
| 総合リスク | **低〜中** |
| 外部からのリモート攻撃 | **ほぼ不可**（バックエンドなし） |
| 実運用上の主なリスク | **QR なりすまし、PC への物理アクセス、LAN 公開** |

**結論:** クラウドサービス型と比べると攻撃面は小さい。ただし **「誰でも QR を印刷でき、PC に触れればデータを改ざんできる」** 設計のため、入退室管理としての **信頼性** には弱点がある。

---

## 診断範囲

| 対象 | 内容 |
|------|------|
| `index.html` | QR スキャン、localStorage、CSV 出力 |
| `js/jsQR.min.js` | 同梱ライブラリ |
| `起動.bat` | Python HTTP サーバー起動 |
| `.gitignore` | 機密データの除外 |

**対象外:** GitHub 上の公開リポジトリ自体、各 PC の CSV ファイル、ブラウザ/OS の脆弱性

---

## 発見事項

### 高 — QR コードのなりすまし（入退室記録の改ざん）

| 項目 | 内容 |
|------|------|
| 内容 | QR の中身は **生徒IDの平文のみ**。署名・有効期限・本人確認なし |
| 再現 | 他人の ID（例: `S001`）を入れた QR を印刷してスキャン |
| 影響 | **来塾/帰宅を不正記録可能** |
| 対策 | QR に HMAC 署名を付ける、PIN 入力、IC カード併用、スタッフ確認 |

> **最も現実的な脅威** である。

---

### 中 — 認証・権限管理なし

| 項目 | 内容 |
|------|------|
| 内容 | ログインなし。アプリを開いた人が全機能利用可能 |
| 再現 | PC の前に座り、DevTools（F12）を開く |
| 影響 | 生徒マスタ閲覧・改ざん、記録削除、偽記録の直接投入 |

DevTools で以下のような操作が可能:

```javascript
// 開発者ツールで実行可能
localStorage.setItem('juku_records_v1', JSON.stringify([...]));
```

| 対策 | PC ログイン・画面ロック、共用 PC 禁止、操作ログの外部保存 |

---

### 中 — Python HTTP サーバーが LAN に公開される可能性

| 項目 | 内容 |
|------|------|
| 内容 | `python -m http.server` はデフォルトで **全インターフェース（0.0.0.0）** にバインド |
| 再現 | 同一 Wi‑Fi 内の別端末から `http://<このPCのIP>:8765/` にアクセス |
| 影響 | **アプリ本体（HTML/JS）の取得は可能**。ただし **localStorage のデータは取得不可**（端末・オリジンごとに独立） |
| 対策 | `python -m http.server 8765 --bind 127.0.0.1` に変更 |

---

### 中 — CSV 数式インジェクション

| 項目 | 内容 |
|------|------|
| 内容 | CSV 出力時、生徒ID を **クォートなし** で出力 |
| 再現 | 生徒ID を `=1+1` や `=cmd|'/c calc'!A0` などで登録 → CSV を Excel で開く |
| 影響 | Excel 起動時に **数式が実行される** 可能性（Excel の設定依存） |
| 対策 | CSV フィールドを `"..."` で囲む。先頭が `=`, `+`, `-`, `@` なら `'` を付与 |

---

### 低 — localStorage 改ざんによる XSS の可能性

| 項目 | 内容 |
|------|------|
| 内容 | 通常 UI 経由の入力は `escapeHtml()` で保護されているが、`records` の `type` フィールド等は **innerHTML に未エスケープ** で描画 |
| 該当箇所 | `${r.type}`、`stay.lastRecord.type` など |
| 再現 | DevTools で localStorage を改ざんし、`<img src=x onerror=alert(1)>` を `type` に設定 |
| 影響 | **同一 PC・同一ブラウザ上** で XSS 可能。外部からのリモート XSS ではない |
| 対策 | すべての動的 HTML 出力で `escapeHtml()` を徹底 |

通常の QR スキャンだけでは `type` は「来塾」「帰宅」のみなので、**日常利用だけなら発生しにくい**。

---

### 低 — カメラ映像の取り扱い

| 項目 | 内容 |
|------|------|
| 内容 | Web カメラ映像は QR 読み取りのみ。外部送信のコードなし |
| リスク | ブラウザのカメラ許可を与えた端末上でのみ利用 |
| 対策 | 共用 PC ではカメラ許可に注意 |

---

### 低 — セキュリティヘッダー未設定

| 項目 | 内容 |
|------|------|
| 内容 | CSP、X-Frame-Options 等なし |
| 影響 | localhost 利用が主なので影響は限定的 |
| 対策 | `<meta http-equiv="Content-Security-Policy" ...>` の追加を検討 |

---

### 情報 — その他

| 項目 | 評価 |
|------|------|
| CDN 依存 | **解消済み**（jsQR 同梱） |
| 秘密情報のハードコード | **なし** |
| `.gitignore` で CSV 除外 | **適切** |
| GitHub 公開 | ソース公開のみ。利用者データは含まれない |
| 3 秒クールダウン | 同一 ID の連続スキャン防止のみ。**別 ID なら連続記録可能** |

---

## 問題なかった項目

- SQL インジェクション（DB なし）
- サーバーサイド RCE（サーバー処理なし）
- セッションハイジャック（セッションなし）
- QR 内容の通常経路での XSS（`escapeHtml` あり）
- カメラ映像の外部送信（該当コードなし）

## リスクマトリクス（まとめ）

| # | 脆弱性 | 深刻度 | 外部攻撃 | 対策優先度 |
|---|--------|--------|----------|------------|
| 1 | QR なりすまし | 高 | 不要（QR 印刷のみ） | ★★★ |
| 2 | 認証なし | 中 | 不可（物理アクセス） | ★★★ |
| 3 | HTTP サーバー LAN 公開 | 中 | 同一 LAN から可能 | ★★☆ |
| 4 | CSV 数式インジェクション | 中 | 不可 | ★★☆ |
| 5 | localStorage 改ざん XSS | 低 | 不可 | ★☆☆ |
| 6 | セキュリティヘッダー不足 | 低 | 限定的 | ★☆☆ |

---

## 推奨対策（優先順）

1. **運用:** 受付 PC にログイン/画面ロック、QR だけで本人確認しない
2. **起動.bat:** `--bind 127.0.0.1` で localhost のみに限定
3. **CSV:** フィールドをクォート、数式インジェクション対策
4. **コード:** すべての innerHTML 出力を `escapeHtml()` 化
5. **将来:** 署名付き QR、PIN、操作ログの外部保存

---

## 補足（診断の限界）

今回は **ローカル単体アプリ** のため、以下は実施していない。

- OWASP ZAP 等による本格的な自動スキャン
- インターネット公開環境（GitHub Pages 等）での外部攻撃
- ソーシャルエンジニアリング、物理セキュリティ

用途が「**塾内の 1 台の PC で使う入退室管理**」であれば、現状でも **一般的な Web アプリよりリスクは低い**。一方で **記録の真正性** を重視するなら、QR なりすましと認証なしが最大の弱点である。
